# 페이지 및 라우트 명세

청약닷컴의 페이지별 상세 명세서입니다. 각 페이지의 목적, 데이터 요구사항, 레이아웃, 인터랙션, 수용 기준을 정의합니다.

---

## 공통 요소

### 하단 내비게이션 바 (모바일)

- **위치:** 하단 고정, 글래스모피즘 표면 (`color-bg-page` @ 80% + `backdrop-blur: 20px`)
- **항목:** 홈 (`/`), 청약 (`/listings`), 뉴스 (`/news`), 검색 (오버레이)
- **활성 상태:** `brand-primary-500` 아이콘 + 라벨, 비활성은 `neutral-400` 사용
- **높이:** 64px + 기기 하단 인셋 (안전 영역)

### 상단 헤더 (데스크톱, ≥1024px)

- **위치:** 상단 고정, 글래스모피즘 표면
- **레이아웃:** 로고 (좌측) | 내비게이션 링크 (중앙) | 검색 아이콘 (우측)
- **내비게이션 링크:** 홈, 청약, 뉴스
- **로고:** 프로젝트 루트의 `logo.svg` 사용

### 통합 검색 (오버레이)

> **상태:** 베타 단계 비공개. UI 숨김 처리, 코드는 보존. 복원 절차는 `docs/beta-launch-deferred-features.md#search` 참고.

- **실행:** 네비게이션 검색 아이콘 클릭 또는 `⌘K` (`Ctrl+K`) 단축키
- **범위:** 청약 (이름, 위치, 시공사) 및 뉴스 (제목, 본문) 전체 검색
- **UI:** 전체 화면 모달 오버레이 (배경 딤), 중앙 패널 (최대 640px)
- **결과:** 유형별 그룹화 (청약, 뉴스), 그룹당 최대 5개 미리보기
- **최근 검색어:** localStorage에 저장, 최대 10개
- **키보드:** `Escape`로 닫기, 열릴 때 입력란 자동 포커스

### 푸터

- **스타일:** 최소한의 디자인, `surface` 배경
- **링크:** 소개, 이용약관, 개인정보처리방침
- **내용:** 저작권 표시, 로고 마크

## 페이지 1: 홈 대시보드

- **라우트:** `/`
- **목적:** 주간 청약 일정과 시장 인사이트를 한눈에 보여주는 화면. "이번 주 청약은 뭐가 있지?"라는 질문에 답하는 첫 화면.

### 섹션

#### 1.1 히어로 / 주요 청약

- 가장 주목할 만한 예정 청약을 강조하는 전체 너비 카드
- 내용: 아파트명, 위치, 주요 일정, 세대수, 시공사명
- 배경: `white` 카드 위에 은은한 그라데이션 오버레이
- CTA: `/listings/[id]`로 연결되는 "상세 보기" 버튼

#### 1.2 이번 주 일정 타임라인

- 가로 스와이프 카드 캐러셀 (모바일) / 그리드 행 (데스크톱)
- 카드별 내용: 아파트명, 위치 요약, 접수 기간, 상태 칩
- 상태 칩: 접수중은 `success`, 마감임박은 `warning`, 접수예정은 `info` 사용
- 접수 시작일 기준 오름차순 정렬
- 빈 상태: "이번 주 예정된 청약이 없습니다" + 전체 목록 링크

#### 1.3 시장 인사이트 카드

- 2열 그리드 (모바일) / 3열 그리드 (데스크톱)
- 카드별 내용: 지표명, 수치, 추세 표시 (증감 화살표 + 백분율)
- 지표: 평균 경쟁률, 이번 달 총 세대수, 인기 지역
- 배경: `surface` 페이지 배경 위 `white` 카드

#### 1.4 전국 실거래가 TOP 5

- 최근 2일 거래 중 거래금액 상위 5건을 순위형 행 리스트로 노출
- 행별 내용: 순위(1~3위 브랜드 강조 / 4~5위 톤다운), 아파트명, 지역(시군구+동), 공급면적이 아닌 전용면적 + 평 병기(`85㎡ (34평)`), 층, 거래일, 거래금액(우측 강조)
- 섹션 부제: "최근 2일 거래 · 거래금액 상위 · 출처: 국토교통부"
- 배열이 비었거나 상위 요청이 실패하면 섹션 전체를 숨김
- 상세 페이지 링크 없음 (실거래 상세/검색 페이지 신설 후 후속 티켓에서 연결)
- 면적 축 주의: 청약 리스팅은 공급면적(`minSupplyArea`/`maxSupplyArea`) 기준, 실거래는 전용면적(`exclusiveArea`) 기준 — 서로 다른 축이므로 라벨 혼동 방지

#### 1.5 최신 뉴스 미리보기

- 최근 뉴스 기사 3개를 간결한 카드로 표시
- 카드별 내용: 카테고리 칩, 제목 (2줄 제한), 날짜
- `/news`로 이동하는 "전체 보기" 링크

### 데이터 요구사항

| 엔드포인트 | 데이터 |
|---|---|
| `GET /main/featured` | 주요 청약 단일 객체 (공급면적 기준) |
| `GET /main/weekly-schedule` | 월~금 일자별 공고 목록 |
| `GET /main/stats` | 이번 달 경쟁률·공급세대 통계 + 인기 지역 |
| `GET /main/top-trades` | 거래금액 상위 최근 실거래 5건 (전용면적 기준) |
| `GET /api/news/latest?limit=3` | 최신 뉴스 기사 3개 |

### 모바일 레이아웃

- 세로 스택: 히어로 → 일정 캐러셀 → 실거래 TOP 5 → 뉴스 미리보기
- 일정 섹션은 스냅 포인트가 있는 가로 스크롤 사용
- 아래로 당겨 새로고침 시 전체 데이터 리로드

### 주요 인터랙션

- 청약 카드 탭 → `/listings/[id]`로 이동
- 뉴스 카드 탭 → `/news/[id]`로 이동
- 일정 캐러셀 가로 스와이프
- 아래로 당겨 전체 대시보드 데이터 새로고침

### 수용 기준

- [ ] 히어로 섹션에 주요 청약의 모든 필수 필드가 렌더링됨
- [ ] 주간 일정에 이번 주 해당 청약이 올바르게 표시됨
- [ ] 상태 칩이 청약 상태별로 올바른 색상 토큰을 사용함
- [ ] 시장 인사이트 카드에 추세 방향과 백분율이 표시됨
- [ ] 실거래가 TOP 섹션에 순위 1~5위가 거래금액·지역·전용면적(평 병기)·층·거래일과 함께 노출되고, 데이터가 비었을 때 섹션이 숨겨짐
- [ ] 4G 모바일 환경에서 2초 이내 로드 (LCP < 2s)

## 페이지 2: 청약 목록

- **라우트:** `/listings`
- **목적:** 필터링 가능한 전체 아파트 청약 목록. 사용자가 청약 기회를 탐색하는 주요 화면.

### 섹션

#### 2.1 고정 필터 바

- **위치:** 헤더 아래 고정 (데스크톱) 또는 상태 바 아래 고정 (모바일)
- **스타일:** `white` 배경에 은은한 하단 그림자
- **필터 (Phase 6 기준 `/apt-sales` 에 바인딩):**

| 필터 | 유형 | 옵션 | 쿼리 키 |
|---|---|---|---|
| 지역 (시/도) | 다중선택 드롭다운, grouped chip pool (수도권/광역시/도) | 전국 17개 시/도 | `region` |
| 상태 | 다중선택 칩 | 접수예정, 접수중, 발표대기, 발표일, 청약완료 | `status` |
| 청약 유형 | 다중선택 칩 | 공공 (public), 민간 (private) | `type` |

- 지역 드롭다운은 팝오버/시트 안에서 grouped slim chip 으로 렌더링합니다
- 다중선택 칩은 활성 시 `Check` 아이콘을 앞에 붙입니다 (색상 전용 금지 규칙)
- 상태/유형 칩은 시각적으로는 slim `h-8` 이지만 `chip-hit-slop` utility 로 44×44 터치 타겟을 유지합니다
- 모바일 필터 아이콘에 활성 필터 개수 배지 표시
- 모바일 시트 footer 는 필터가 즉시 적용되므로 "적용" 대신 "닫기"를 사용합니다
- "초기화" 버튼은 모든 필터(상태/유형/지역)를 한 번에 비웁니다

**로드맵 (백엔드 미지원으로 현재 미구현):**

| 필터 | 차단 사유 |
|---|---|
| 구/군 | `regionCode` 가 시/도 단위까지만 제공 |
| 공급 유형 (특별/일반) | 쿼리 파라미터 없음. 공급 분해는 상세 엔드포인트에서만 노출 |
| 시공사 | 쿼리 파라미터 없음. `constructorName` 은 아이템 응답에만 포함 |

#### 2.2 정렬 컨트롤

- **옵션:** 날짜순 (최신순, 기본값), 날짜순 (오래된순), 인기순 (조회수)
- **스타일:** 결과 위 우측 정렬 인라인 드롭다운

#### 2.3 청약 카드

- **레이아웃:** 전체 너비 카드 스택 (모바일) / 2열 그리드 (데스크톱)
- **카드 내용:**
  - 아파트명 (헤드라인)
  - 위치 (시/도 + 구/군 + 동)
  - 일정: 접수 기간 (시작일–종료일)
  - 상태 칩 (접수중 / 접수예정 / 마감)
  - 시공사명
  - 세대수 및 면적 범위 (예: "84㎡ ~ 114㎡")
- **카드 스타일:** `white` 배경, `8px` 반경, 테두리 없음 (디자인 시스템 준수)
- **인터랙션 상태:** 호버/프레스 시 배경이 `surface`로 전환

#### 2.4 페이지네이션

- SEO를 위한 페이지 기반 내비게이션 (무한 스크롤 아님)
- 표시: 이전 / 페이지 번호 / 다음
- URL에 현재 페이지 반영: `/listings?page=2`
- 페이지당 20개 항목

### 데이터 요구사항

| 엔드포인트 | 데이터 |
|---|---|
| `GET /apt-sales` | 페이지네이션된 청약 목록. 라우트 파일은 얇은 Server Component 셸이고, `SubscriptionListClient` 가 `useQuery(aptSalesQueryOptions(request))` + `keepPreviousData` + 60s `staleTime` 로 데이터 fetch 를 소유해 세션 내 재방문을 즉시 렌더합니다. |
| **쿼리 파라미터 (바인딩 완료)** | `status[]`, `houseDetailType[]`, `regionCode[]`, `page`, `size` |
| **쿼리 파라미터 (로드맵)** | `district` (구/군), `supply` (공급유형), `builder` — 백엔드 미지원. API 도착 전까지 UI 노출 보류 |

지역 옵션은 orval 이 생성한 `ItemRegionCode` enum 에서 가져오며 별도
`/filters/*` 엔드포인트를 호출하지 않습니다. 구 문서에 남아있던
`/api/filters/regions`, `/api/filters/builders` 표기는 무효입니다.

### 모바일 레이아웃

- 필터 바가 필터 아이콘 + 활성 필터 개수 배지가 있는 단일 행으로 축소
- 필터 아이콘 탭 시 전체 화면 필터 시트 (하단 드로어) 열림
- 카드는 전체 너비, 16px 세로 간격으로 분리
- 스크롤 중에도 고정 필터 바 유지

### 주요 인터랙션

- 필터 선택 → 결과 즉시 갱신 (별도 제출 버튼 없음)
- 카드 탭 → `/listings/[id]`로 이동
- 페이지 변경 → 상단으로 스크롤, URL 갱신
- 필터 상태가 URL 쿼리 파라미터에 유지되어 공유 가능

### 수용 기준

- [ ] 3가지 필터 유형이 각각 및 조합으로 동작함
- [ ] URL에 모든 활성 필터와 현재 페이지가 반영됨
- [ ] 카드에 모든 필수 필드가 올바른 상태 칩 색상과 함께 표시됨
- [ ] 필터 조건에 맞는 청약이 없을 때 빈 상태가 표시됨

## 페이지 3: 청약 상세

- **라우트:** `/listings/[id]`
- **목적:** 하나의 청약에 대한 완전한 정보 제공. 사용자가 자격 요건과 다음 단계를 판단하는 의사결정 페이지.

### 섹션

#### 3.1 단지 개요 헤더

- 아파트명 (디스플레이 헤딩)
- 위치 정보 (시도/구군/동); `supplyAddress` 가 있으면 위치 라인 아래에
  보조 텍스트로 노출
- 시공사명 (`constructorName`); `businessEntityName` 이 시공사와 다르면
  `사업주체 …` 보조 라인 추가
- 주요 통계 행: 총 세대수, 면적 범위, 예상 분양가 범위, 접수기간,
  그리고 `moveInMonth` 가 있으면 입주예정 (`YYYY년 M월` 포맷)
- 상태 칩 (대형)
- 작성자 또는 편집자 정보 표시 금지

#### 3.2 일정 타임라인

- 최대 7단계 세로 타임라인. `map-apt-sales-detail.ts` 매퍼가 백엔드의
  `ScheduleSection` + `moveInMonth` 를 아래 순서로 접고, 날짜가 null 인
  단계는 제외하여 빈 줄이 생기지 않도록 합니다:
  1. 모집공고 (`schedule.announcementDate`)
  2. 특별공급 접수 (`schedule.specialSupply`)
  3. 일반공급 1순위 (`schedule.firstRankLocal` — `firstRankGyeonggi` / `Other` 는 주 타임라인에선 생략)
  4. 일반공급 2순위 (`schedule.secondRankLocal`)
  5. 당첨자 발표 (`schedule.winnerAnnouncementDate`)
  6. 계약 기간 (`schedule.contract`)
  7. 입주 예정 (`announcement.moveInMonth` 을 `yyyy-MM-01` 로 확장)
- 각 단계에 날짜와 current/past/future 상태 표시 (오늘 기준)
- 현재 단계는 `primary` 강조색으로 하이라이트
- 지난 단계는 `text-low`, 향후 단계는 `text-mid` 사용
- dot 간 세로 연결선은 비-마지막 아이템마다 `top-3`에서 `-bottom-0.5`
  까지 그려 다음 dot 에 정확히 닿게 한다(이전 구현은 `bottom-0`이라
  중간에 끊김). 마지막 dot 아래로는 선이 흐르지 않음.
- 섹션은 `bg-bg-card rounded-lg p-6` 카드로 감싸 단일 primary 블록으로
  읽히게 한다. 공급내역/경쟁률/당첨가점/특공 섹션은 자체 내부 컨테이너
  (카드 또는 `rounded-lg` 테이블)가 이미 surface 역할을 하므로 페이지
  캔버스 위에 평탄하게 흘린다.

#### 3.3 공급 내역 — 평형별 카드

- `models[]` 엔트리 1개 = 카드 1장 (평형 단위)
- 카드 헤더는 `formatHouseType` 으로 정규화한 주택형(`059.9400A → 59A`)
  만 노출 — 내부 식별자인 `modelNo` 는 더 이상 사용자에게 보여주지 않음
- 카드 내용: 공급면적(㎡), 일반공급 세대수, 특별공급 세대수, 특공 세분
  유형별 세대수(다자녀/신혼부부/생애최초/노부모부양/기관추천/기타/이전기관/청년/신생아 중 값이 있는 것만), 최고 분양가(억/만 포맷)
- 레이아웃: `bg-bg-card` (흰) 카드를 페이지 `bg-bg-page` 캔버스 위에
  직접 배치. surface tier 차이만으로 lift 표현(DESIGN.md §8 의 ambient
  shadow 는 floating 요소 전용 룰을 준수해 카드에는 그림자를 두지 않음).
  sm+ 2열 그리드, 모바일 1열
- 카드 내부 특공 세분 칩은 `bg-bg-sunken` (카드 표면 위 inset 패턴)
- 헤더의 면적/가격 범위는 models 전체의 min~max 를 집약해 스크롤 없이도 전체 스펙 파악

#### 3.4 자격 요건 간편 확인 *(보류)*

- 해당 백엔드 엔드포인트가 아직 없어 보류. 소득/자산/거주 요건이 노출되면 재개합니다.

#### 3.5 관련 뉴스 *(보류)*

- `/apt-sales/{id}` 응답 범위 밖입니다. 뉴스 연계 전용 엔드포인트 배포 이후 재개.

#### 3.6 공식 링크

- 모집공고 링크 ← `announcement.announcementUrl` (해당 API 필드 자체가
  모집공고문 URL — 이전 코드는 이를 청약홈으로 잘못 매핑했음)
- 시공사 공식 사이트 ← `announcement.homepageUrl`
- 청약홈 신청 링크 ← *(보류)* API 에 청약홈 전용 딥링크 필드가 없어
  `applyHomeUrl` 은 딥링크 룰 도입 전까지 `undefined` 로 둠
- 문의 전화 ← `announcement.inquiryPhone` 을 `tel:` 링크 카드로 노출
  (라벨에 표시 번호 포함)
- 순서는 고정: **모집공고**가 항상 사이드바 1번(최상단). 2번 = 청약홈
  신청(가능할 때), 3번 = 시행사, 4번 = 문의. 시각 primary 는 가용성에
  따름 — `applyHomeUrl`이 undefined 인 현재는 1번 자리 모집공고가
  primary(brand-primary) 버튼. 향후 청약홈 딥링크가 도입되면 청약홈이
  2번 자리에 들어와 primary 색을 가져가고 모집공고는 1번 자리에서
  secondary 로 내려간다. §11.2 "view 당 primary 하나" 룰 유지.

#### 3.7 경쟁률 (Competitions)

- `competitions[]` 행: 주택형 × 순위 × 거주지역
- 컬럼: 주택형, 순위, 거주지역, 공급세대, 접수건수, 경쟁률
- 주택형은 `formatHouseType` 정규화 표기(`059.9400A → 59A`),
  같은 주택형 연속 행은 첫 행에 `rowSpan` 으로 펼쳐 평형별 그룹으로 읽힘
- 행 구분은 **그룹 단위 zebra**: 같은 주택형 그룹 내 모든 행은 같은
  배경(`bg-bg-card` 또는 `bg-bg-page`)을 공유하고, 그룹 사이에서만
  배경이 교차한다. 평형 경계가 어떤 구분선도 없이 인지된다(DESIGN.md
  §11.5 No-Line 룰을 어기지 않으면서 zebra만으로 그룹화 표현). 외부
  래퍼는 `rounded-lg overflow-hidden`, 헤더는 `bg-bg-sunken`.
- `rateDisplay` 는 서버가 포맷한 값 (`"15.00"`, `"(N세대 부족)"`, `"미달"`) 을 그대로 노출; `isShortage` true 행은 `warning-600` 으로 강조
- `competitions[]` 가 비어있으면 (집계 전) 섹션 자체를 렌더하지 않음

#### 3.8 당첨가점 (Winner Scores)

- `winnerScores[]` 행: 주택형 × 거주지역
- 컬럼: 주택형, 거주지역, 최저/평균/최고 (서버가 내려준 `*Display` 문자열, 없으면 `-`)
- §3.7 와 동일 형식: `formatHouseType` + 평형별 `rowSpan` 그룹핑 +
  zebra row striping + `rounded-lg` 외부 래퍼
- 빈 배열이면 섹션 숨김

#### 3.9 특별공급 신청현황 (Special Supply Status)

- `specialSupplies[].categories[]` 를 (주택형 × 유형) 단위로 flatten
- 컬럼: 주택형, 유형, 배정, 해당지역, 기타경기, 기타지역, 합계
- §3.7 와 동일(formatHouseType + rowSpan + zebra)에 더해, 주택형 컬럼은
  `position: sticky left-0 z-10` 로 좌측 고정하고 행 zebra 배경을
  그대로 받아 모바일 가로 스크롤 시에도 행 정체성이 유지
- 기관추천/이전기관 행의 지역 컬럼은 `-` (API 보장)
- 빈 배열이면 섹션 숨김
